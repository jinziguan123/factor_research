# 因子手册「图表解读」交互 Mock 设计文档

> 日期：2026-04-22
> 状态：已确认方案，待实施
> 作者：Codex（与用户共同收敛）

## 0. 一句话目标

在前端 `因子手册` 页面为核心评估图新增“典型好图 vs 典型坏图”的**可交互 mock 对照**，让读者能把“判别规则”直接映射到真实评估图。

## 1. 第一性原理与范围

### 1.1 原始问题

用户看到评估页图表后，常见痛点不是“看不到图”，而是“看到了也不知道怎样算好/坏”。  
所以产品核心不是再加一份静态说明，而是建立“**图形形态 ↔ 判别结论**”的低摩擦学习闭环。

### 1.2 本期范围（MVP）

- 页面：`/docs/factor-guide`
- 首张图：`IC 衰减曲线`
- 呈现方式：同屏并排“好图/坏图”+ 上方判别清单
- 数据来源：前端 fixture 文件（不依赖后端）
- 图表能力：复用现有图表组件，不新造图表内核

### 1.3 非目标（本期不做）

- 不改评估详情页真实图表逻辑
- 不新增后端 seeded eval / 特殊 API
- 不做交互式箭头标注 tooltip 系统
- 不一次性覆盖所有图，先用一张图验证模式

## 2. 调研结论（上下文重建）

1. `AppSidebar` 菜单已挂载 `因子手册 -> /docs/factor-guide`。  
2. `FactorGuide.vue` 当前为完整静态内容页，不是占位页。  
3. 现有图表组件（`IcDecayChart.vue`、`GroupReturnsChart.vue`、`TurnoverChart.vue`、`EquityCurveChart.vue`）都接受轻量 `dates + values`/`g1..gN` 结构，适配 mock 成本低。  
4. 评估详情页核心图已稳定，优先值得做解读的是：IC 衰减、分组净值、换手率、多空净值。  
5. 文风基准来自 `docs/factor-engineering-basics.md` 末尾“附”：以“指标-现象-结论-建议”为主的诊断式表达。

## 3. 已确认决策

1. 第一张解读图：`IC 衰减曲线`
2. mock 数据来源：`独立 fixture 文件`
3. 页面排版：`上文下图`（先解读，后图表）
4. 对比方式：`同屏并排 好图 vs 坏图`
5. 文案组织：`Checklist 判别清单`

## 4. 方案比较与选择

### 方案 A（选定）：复用现有图表组件 + docs fixtures + 手册页解读块

- 优点：开发路径最短；视觉语义与评估页一致；维护成本最低
- 风险：教学交互形态受现有组件边界约束

### 方案 B：为手册单独做简化图表组件

- 优点：可定制教学动画/标注
- 代价：维护两套图表实现，长期易漂移

### 方案 C：直接嵌评估详情页图表容器

- 优点：复用最深
- 代价：手册页和评估页耦合过高，后续改动风险大

### 选择理由

以最短路径验证价值优先，先确认“解读模式本身”是否有效，再决定是否投资更重的教学交互能力。

## 5. 页面与交互设计

## 5.1 信息结构（MVP）

1. 模块标题：图表解读（IC 衰减）
2. 判别清单（Checklist）：
   - 好图应满足什么
   - 坏图常见症状
   - 实盘含义与调仓建议
3. 图表区：左右两列
   - 左：典型好图（绿色标签）
   - 右：典型坏图（红色标签）
4. 图下注释：强调“此处为教学 mock，不代表真实回测结果”

## 5.2 阅读路径

用户先读清单，再看并排图对照，最后带着规则回到评估详情页核对真实图。这比“先看图再猜意思”更省认知负担。

## 6. 技术设计

### 6.1 数据层（fixtures）

新增 docs 侧 fixture 文件，集中维护“好/坏”两套 IC 衰减样例：

- `good`: 峰值合理、衰减平滑、半衰期较长
- `bad`: 峰值尖刺、快速衰减或反号、半衰期过短

数据结构直接兼容 `IcDecayChart` 现有 props：

```ts
{
  ic: Record<string, { dates: string[]; values: (number | null)[] }>,
  rankIc: Record<string, { dates: string[]; values: (number | null)[] }>
}
```

### 6.2 组件层

- 复用：`frontend/src/components/charts/IcDecayChart.vue`
- 新增：因子手册页内的“解读卡片 + 双图容器”（页面内局部组件或抽离 docs 组件）
- 不改：评估页 `EvalDetail.vue` 的图表渲染逻辑

### 6.3 样式层

- 使用现有 Naive UI 卡片体系
- 桌面端双列，移动端自动单列换行
- 使用明确视觉标签区分好/坏（颜色与文案都区分）

## 7. 风险与缓解

1. 风险：mock 与真实图视觉语义不一致  
缓解：强制复用 `IcDecayChart`，避免“教学图和生产图不是一回事”。

2. 风险：文案漂移或过长，读者读不完  
缓解：Checklist 每条一句话，保持“可扫描”。

3. 风险：后续扩图时结构混乱  
缓解：fixture 采用按图表类型分文件/分对象组织，先定统一 schema。

## 8. 验收标准（MVP）

- `FactorGuide` 页面可见“IC 衰减图表解读”模块
- 同屏可交互查看“好图 vs 坏图”两张 `IcDecayChart`
- 解读文案为 checklist 形式，且每条能映射到图上可观察现象
- 页面在桌面与移动端均可正常展示（无严重布局错乱）
- 不引入后端请求依赖（离线可展示）

## 9. 后续扩展顺序

在本模式验证通过后，按以下优先级扩展：

1. 分组累计净值
2. 多空净值曲线
3. 换手率
4.（可选）IC/RankIC 序列

每新增一张图都沿用同一模板：`Checklist + 好坏对照 + 教学声明`，避免页面风格碎片化。
